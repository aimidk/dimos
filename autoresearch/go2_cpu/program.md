# go2_cpu autoresearch

Karpathy-style autonomous research loop adapted for DimOS: minimize the CPU
footprint of a Go2 basic-blueprint replay run. Modeled on
[karpathy/autoresearch](https://github.com/karpathy/autoresearch).

## Setup

This directory is tracked on the `ruthwik/autoresearch` branch, so each
experiment is a real git commit and keep/revert uses `git reset --hard`
(Karpathy's original pattern). Setup steps:

1. **Confirm branch**: you should be on `ruthwik/autoresearch` or a sub-branch
   like `ruthwik/autoresearch/expN` created from it. Do not run the loop on
   `dev` or `main`.
2. **Read the in-scope files** for full context (the directory is small):
   - `README.md` — goal, metric, known hot paths.
   - `eval.py` — runs the replay under `time`, samples the process tree with
     psutil, parses `user+sys`, writes a `BenchmarkResult` JSON. **Do not
     modify.** This is the ground-truth metric harness.
   - `optimizations.py` — the one file you edit. Returns
     `{cli_args, env, startup_code}`, which `eval.py` splices into the replay
     subprocess.
   - `_patches/sitecustomize.py` — auto-generated by `optimizations.py` on
     each run. Don't hand-edit.
3. **Verify the replay dataset exists**: `ls data/unitree_go2_bigoffice/`
   from the dimos repo root must show `lidar/`, `odom/`, `video/`. If not,
   the data lives at `data/.lfs/unitree_go2_bigoffice.tar.gz` — tell the human
   to extract it.
4. **Initialize `results.tsv`** with the header row if it doesn't exist:
   `commit\tscore\treal\tstatus\tdescription`. **Do not commit `results.tsv`**
   — it's in `.gitignore`. The baseline row is written after the first run.
5. **Record a clean baseline**: before any experiments, confirm
   `optimizations.py::apply()` is the baseline you want (empty `{}` for a
   true zero-optimization baseline, or the current state if you're continuing
   a prior session). Commit it: `git commit -am "baseline"`.
6. **Confirm and go.**

## Experimentation

Each experiment runs the full Go2 replay blueprint for a **fixed 300 s
wall-clock budget** (configurable via `--timeout`). You launch it as:

```bash
bash run.sh                      # full 5-minute run
bash run.sh --timeout 60         # short iteration (noisier, use for sanity)
bash run.sh --skip-profile       # skip the py-spy profile step
```

**What you CAN do:**
- Edit `optimizations.py` only. The `apply()` function returns a dict with:
  - `cli_args` — extra CLI flags inserted before the `run` subcommand (e.g.
    `--n-workers=2`, `--log-level=WARNING`).
  - `env` — extra env vars for the subprocess (e.g. `OMP_NUM_THREADS=1`).
  - `startup_code` — Python code written to `_patches/sitecustomize.py` and
    injected via `PYTHONPATH`. This runs before `dimos` imports, so you can
    monkey-patch module-level constants, thread-pool sizes, poll timeouts,
    etc.

**What you CANNOT do:**
- Modify `eval.py` — it's the ground-truth harness.
- Install new packages or add dependencies.
- Modify DimOS source files to "fix" things. If a knob is hardcoded, patch it
  via `startup_code` monkey-patching, not by editing the `dimos/` tree.
- Weaken the metric (e.g. skip the run, fake the timing output).
- Commit to `main` or `dev`. Stay on `ruthwik/autoresearch` (or a
  sub-branch). Never force-push.

**Goal: minimize `SCORE = user + sys` CPU seconds** over the replay window,
without breaking the LCM pipeline.

## Known harness limitations (read before optimizing)

These are real gaps — work around them, don't exploit them:

1. **Replay loops forever.** The blueprint has no `--exit-on-eof` path, so
   every run ends in a 300 s timeout. `SCORE` is "CPU spent in 300 s of wall
   time," not "CPU per unit of work." A change that makes replay run *faster*
   can process more messages and raise `SCORE` while actually being an
   improvement. Watch `peak_cpu_pct`, `avg_threads`, and context switches for
   sanity. **Do not optimize for `SCORE` alone without sanity-checking the
   secondaries.**
2. **Validation is a stub.** `validation: PASS` is printed on any clean exit.
   There is no real LCM-message-count or payload diff against a baseline
   recording yet. **You can silently break the pipeline and the harness will
   not notice.** Always manually check that your patch is semantically
   plausible — e.g. dropping an RPC thread pool to 0 workers would still
   "pass."
3. **Single run, no variance gate.** One sample per candidate. A 5% score
   delta is likely noise. Aim for ≥15% improvements, or run the same config
   3× manually and average.
4. **Profile is a stub.** `run_profile()` writes a placeholder unless `py-spy`
   is on PATH. If you want real profiling data, run manually:
   `py-spy record -o profile.svg -d 60 -- uv run dimos --replay --viewer=none
   --replay-dir=unitree_go2_bigoffice run unitree-go2`
   and read `profile.svg` in a browser.

Gaps #1 and #2 are the ones that most limit signal quality. If you find
yourself chasing small deltas, stop and talk to the human about closing these
gaps before iterating further.

## Output format

Each `eval.py` run prints a summary like this:

```
==================================================
SCORE: 697.47
VALIDATION: PASS (basic (process exited cleanly))
REAL: 300.44
USER: 399.16
SYS: 298.31
PEAK_CPU_PCT: 0.0
PEAK_MEMORY_MB: 2599.8
AVG_THREADS: 351.9
IO_READ_MB: 10527.6
IO_WRITE_MB: 0.0
==================================================
SAVED: .../results/run_20260330_181832.json
```

Extract the key metric from the log:

```bash
grep "^SCORE:" results.txt
```

And the JSON (authoritative) from `results/run_*.json`.

## Logging results

Append one tab-separated row per experiment to `results.tsv` (TAB, not comma
— commas break descriptions). `results.tsv` is gitignored — do NOT commit it.

```
commit    score   real    status    description
a1b2c3d   697.47  300.44  keep      baseline (empty apply())
b2c3d4e   612.10  300.12  keep      LCM loop timeout 50→200 ms
c3d4e5f   690.80  300.08  discard   --n-workers=1 (no delta)
d4e5f6g   0.00    0.00    crash     startup_code typo
```

Columns:
1. `commit` — short git hash of the experiment commit (7 chars)
2. `score` — SCORE from the run (`0.00` for crashes)
3. `real` — wall time (`0.00` for crashes)
4. `status` — `keep`, `discard`, or `crash`
5. `description` — terse one-liner of what was tried

## The experiment loop

Because this directory is tracked by git, keep/revert uses git commits
directly (Karpathy's original pattern).

**LOOP FOREVER:**

1. Look at the git state: `git log --oneline -20` and read the last few
   `results.tsv` rows + `results/run_*.json` entries to remember what has
   been tried.
2. Form one hypothesis. Prefer hypotheses grounded in a profile
   (`profile.svg` / `profile_output.txt`) or in the "Known hot paths" list in
   `README.md`. Avoid random hyperparameter jiggling.
3. Edit `optimizations.py` to implement the hypothesis.
4. `git commit -am "<one-line hypothesis>"`.
5. Run: `bash run.sh --skip-profile`.
6. Grep the score: `grep "^SCORE:" results.txt`.
7. If the grep is empty, the run crashed or hung: `tail -n 80 results.txt`
   for the stack trace. If the fix is obvious (typo, missing import), fix
   and re-run (amend the commit with `git commit -a --amend --no-edit`).
   Otherwise `git reset --hard HEAD~1` (revert the bad commit), log `crash`
   in `results.tsv`, move on.
8. If `SCORE < best_score`:
   - Sanity-check the secondary metrics (AVG_THREADS, PEAK_CPU_PCT,
     IO_READ_MB) haven't collapsed to zero (harness limitation #2 means a
     broken pipeline can still "PASS").
   - If sane: the branch already advanced via the commit in step 4. Log
     `keep` in `results.tsv`.
9. If `SCORE >= best_score`: `git reset --hard HEAD~1` to discard the
   candidate commit. Log `discard` in `results.tsv`.
10. Go to 1.

**Never** `git reset --hard` past the baseline commit. **Never** force-push.

**Timeout per experiment:** ~5 min + a few seconds for startup/eval overhead.
If a single run exceeds 10 minutes, kill it and treat it as a crash.

**NEVER STOP.** Once the loop has begun, do not pause to ask "should I keep
going?". The human is asleep or away and expects you to iterate indefinitely
until manually stopped. If you run out of ideas, re-read the "Known hot
paths" in `README.md`, re-read `dimos/protocol/service/lcmservice.py` and
`dimos/protocol/rpc/pubsubrpc.py`, look at
`dimos/robot/unitree/go2/connection.py` for daemon threads, and combine
previous near-misses. The loop runs until interrupted.
